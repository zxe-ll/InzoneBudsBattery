# InzoneBudsHidProbe

Sony INZONE Buds（WF-G700N）のUSBトランシッターから送られるHID Input Reportを観測し、左右イヤホンと充電ケースのバッテリー残量を検証するWindows向けコンソールアプリです。Dalamudプラグインへ組み込む前の実機検証を目的としています。

## 現在のスコープ

- VID `054C` / PID `0EC2` の全HIDインターフェースを列挙
- 開けるインターフェースをすべて共有アクセスで監視
- Input Reportを非同期で読み取り、既定で16進表示
- 短いHID読取タイムアウトを正常な待機として扱い、数分間隔のレポートを継続待受
- `02 12 04` から始まるレポートのL/R/Caseを解析
- 値を`0～100`に検証
- 実機から推定したチェックサムを表示（不一致だけでは棄却しない）
- 切断後の再スキャンと再接続
- Ctrl+Cによる安全な終了

Dalamud UI、通知、DTRバーはまだ実装しません。実機でHID取得方法が成立すると確認した後に進めます。

## 必要環境

- Windows 10/11 x64
- .NET 8 SDK
- Sony INZONE Buds USBトランシッター（VID `054C`, PID `0EC2`）

HIDアクセスには `HidSharp 2.1.0` を使用しています。アプリはWindowsのHIDデバイスを共有可能な形で開くよう試みますが、INZONE Hubやドライバー側の排他動作によって失敗する可能性があります。

## ビルドと実行

PowerShellでこのディレクトリから実行します。

```powershell
dotnet restore .\InzoneBudsHidProbe.slnx
dotnet build .\InzoneBudsHidProbe.slnx -c Release --no-restore
dotnet run --project .\InzoneBudsHidProbe.csproj -c Release
```

終了は `Ctrl+C` です。受信した全レポートの16進表示を止める場合は `--no-raw`、再スキャン間隔を変える場合は `--scan-seconds 5` のように指定できます。

バッテリーレポートはトランシッターから自発的に送信されます。起動直後に表示されないことは異常とは限りません。イヤホン使用中は3～5分程度、ケース収納時は比較的短い間隔で届くことがあります。

## 期待するログ

```text
INZONE Buds transmitter found (3 HID interface(s)).
Waiting for battery report...

Battery report received:
Left:  82%
Right: 79%
Case:  64%
Checksum: match
Received: 2026-07-15 10:24:31 +09:00
```

各インターフェースのデバイスパスとInput Report長もログに出ます。Hubとの競合や正しいインターフェースの特定に使用してください。

## 実機検証手順

### A. INZONE Hubを終了した状態

1. タスクトレイを含めてINZONE Hubを完全終了する。
2. タスクマネージャーでINZONE Hub関連プロセスが残っていないことを確認する。
3. トランシッターを接続し、イヤホンが利用可能な状態にする。
4. Probeを起動し、検出されたインターフェース数、開けたパス、Report長を保存する。
5. 最大5分程度待つ。イヤホンをケースへ収納／取り出す操作も試す。
6. バッテリーレポートのL/R/CaseをINZONE Hubで後から確認できる値と比較する。
7. トランシッターを抜き、例外終了しないことを確認する。
8. 再接続し、数秒後にインターフェースが再度開かれることを確認する。
9. Ctrl+Cで正常終了することを確認する。

### B. INZONE Hubを起動した状態

1. INZONE Hubを起動し、デバイス画面が表示できることを確認する。
2. Probeを起動し、パターンAと同じ項目を記録する。
3. すべてのパスで `possibly in use` になるか、一部だけ開けるかを確認する。
4. 最大5分程度待ち、HubとProbeのL/R/Caseが一致するか確認する。
5. Hub起動中にトランシッターの切断・再接続を行い、両方が復帰するか確認する。
6. Hubを終了／再起動し、Probe側のログやレポート内容が変化するか確認する。

比較時は、ケース値が最後にイヤホンと通信した時点の古い値である可能性を考慮してください。

## 問題がある場合の調査

- **デバイスが0件:** WindowsのデバイスマネージャーでハードウェアIDに`VID_054C&PID_0EC2`があるか確認します。USBポートの変更も試します。
- **検出するが全パスを開けない:** Hubを完全終了して比較します。管理者実行で結果が変わるかは診断材料になりますが、最終アプリで管理者権限を必須にはしない方針です。
- **開けるがレポートが来ない:** 5分以上待ち、ケースへの収納／取り出しを試します。16進ログとパス、Report長を保存します。
- **`02 12 04`が見つからない:** 全インターフェースの生ログを比較し、先頭に追加バイトが付く環境差がないか調べます。
- **チェックサム不一致:** 現段階では値を表示します。実機ログを複数回集め、式が常に成立するか確認してから必須検証に変更します。
- **Hubと共存不可:** 別インターフェース、ネイティブWindows HID APIの共有オープン、外部ヘルパー＋Named Pipeの順に検討します。Hubのプロセスメモリ読み取りは行いません。

## 次の段階へ進むために保存する情報

- Hub終了時／起動時それぞれの完全な起動ログ
- HIDインターフェース数、各デバイスパス、Input Report長
- バッテリーレポートを最低3～5件（個人情報を含まないことを確認）
- 同時刻のINZONE Hub表示値
- 抜き差し後の復帰ログ
- Ctrl+Cで停止できたか

この結果で共有アクセス、対象インターフェース、64バイト配置、チェックサムを確定し、その後に読み取り処理を再利用可能なサービスへ分離してDalamudプラグインへ移植します。

## 確認済みの実機情報

2026-07-15にINZONE Hubを終了した状態で、5個のHIDインターフェースを検出しました。Input Report長は`62 / 2 / 64 / 0 / 8`で、バッテリーレポートは64バイトの`col03`から受信しました。

受信したレポートでは右100%、左100%、ケース93%、`byte[19] = 0x36`でした。その後、INZONE Hub起動中にセッションフィールドと`byte[19]`が連動して変化する複数のレポートを取得し、次の式が一致することを確認しました。

```text
byte[19] = sum(byte[5] ... byte[18]) mod 256
```

以前の資料にあった`右 + 左 + 117`は、一部の固定フィールドを定数化して推定された可能性があります。現実装ではレポートフィールド全体を使う式へ修正しました。引き続き安全のため、チェックサム不一致だけではレポートを棄却しません。

### INZONE Hubとの共存確認

2026-07-15にINZONE Hub起動中でも、Report長0の`col04`を除く4インターフェースを同時に開けることを確認しました。`col03`から右100%、左100%、ケース93%のレポートを受信でき、`possibly in use`は発生しませんでした。

Hub起動中のトランシッター抜き差しでは、全リーダーが安全に停止し、切断検出から約6秒後に5インターフェースを再検出しました。4インターフェースを再度開き、その約3秒後に`col03`からバッテリーレポートを受信しました。Probe側の自動復帰は確認済みです。

同時刻のINZONE Hub画面も右100%、左100%、ケース93%で、Probeの値と一致しました。トランシッターの抜き差し後はHub側も操作可能な状態へ復帰し、Hubを終了して再起動した後もHubとProbeの両方が正常に動作しました。したがって、現在のHidSharp方式はINZONE Hubと共有利用できることを実機で確認済みです。
